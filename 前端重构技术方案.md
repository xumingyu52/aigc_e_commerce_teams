# 前端重构技术方案

  
> 适用范围：现有双前端架构迁移至 Next.js + React + HeroUI v3  


---

## 一、方案背景与目标

当前项目“智创电商营销系统”采用双前端架构：

- `frontend/`：Next.js 应用，当前承载现代化登录认证界面
- `gui/templates/`：Flask 模板页面，当前承载主要业务功能界面

现状技术形态如下：

- `frontend/` 使用 `Next.js 14.1.0`、`React 19.0.0`、`TypeScript 5.3.3`
- 当前 UI 体系为 `Tailwind CSS 3.4.17` + `Radix UI` + 自建 `components/ui/*`
- `gui/templates/` 业务页面依赖 `jQuery`、`Bootstrap`、`Font Awesome`、`ECharts`、`Marked.js`、`SheetJS`

本次重构的目标不是单纯“重写界面”，而是完成前端技术体系迁移：

1. 将 `gui/templates/` 的业务页面迁移到 `frontend/` 的 Next.js App Router 体系。
2. 将原有 `shadcn/ui / Radix UI + 自定义 UI` 方案统一收敛为 `HeroUI v3`。
3. 保留现有业务术语、接口命名、路由语义、页面职能和后端服务边界。
4. 建立文件级目录结构、组件规范、状态管理规范、样式规范、路由规范和 API 访问规范。
5. 在迁移过程中体现“旧模板页面 -> 新 React 页面”的映射与兼容关系。

---

## 二、现状架构与迁移关系

### 2.1 当前服务划分

根据现有方案与 `main.py` 分析，系统启动 4 个服务：

| 服务 | 端口 | 文件 | 职责 |
|------|------|------|------|
| Next.js 前端 | 3000 | `frontend/package.json` | 登录页面 |
| Login Server | 3002 | `gui/login_server.py` | 登录 API |
| AIGC Server | 5000 | `gui/aigc_server.py` | 主业务服务 |
| App Server | 5001 | `gui/app.py` | 数据看板服务 |

### 2.2 当前前端结构

当前前端分为两部分：

- `frontend/app/page.tsx` 等文件提供登录页与少量现代化界面
- `gui/templates/*.html` 提供首页、数据看板、商品管理、AI 工具、营销模块、分析模块、配置模块

### 2.3 重构后的总体关系

最终技术方案采用“单前端承载业务页面，后端纯接口化”的迁移方向：

- 原 `gui/templates/*.html` 页面迁移到 `frontend/app/**/page.tsx`
- 原模板页内联脚本、jQuery 逻辑拆分到：
  - `components/**`
  - `lib/api/**`
  - `lib/hooks/**`
  - `lib/stores/**`
  - `lib/types/**`
- 原 `Bootstrap + Font Awesome + jQuery DOM 操作` 替换为：
  - `HeroUI v3`
  - `Lucide React`
  - React 组件化与 Hook 化实现
- 原 `Flask 模板渲染页面` 改为 `Next.js App Router` 路由页面
- 后端继续保留：
  - `Login Server:3002`
  - `AIGC Server:5000`
  - `App Server:5001`

## 三、最终确定的技术选型

### 3.1 核心技术栈

最终确定的前端技术栈为：

- `Next.js 14`
- `React 19`
- `TypeScript`
- `Tailwind CSS v4`
- `HeroUI v3`

对应迁移关系如下：

| 维度 | 当前 | 最终方案 |
|------|------|----------|
| 页面承载 | Flask 模板 + Next.js 登录页 | Next.js App Router 统一承载 |
| UI 组件库 | shadcn/ui / Radix UI / Bootstrap | HeroUI v3 |
| CSS 体系 | Tailwind CSS v3.4.17 + Bootstrap | Tailwind CSS v4 + HeroUI styles |
| 图标 | Lucide React + Font Awesome | Lucide React |
| 图表 | ECharts CDN | `echarts` + `echarts-for-react` |
| Markdown 渲染 | Marked.js | `react-markdown` |
| 状态共享 | jQuery 局部状态 / 页面脚本变量 | React Hooks + SWR + Zustand |
| 表单 | 原生 / jQuery | HeroUI 表单组件 + `react-hook-form` + `zod` |
| 上传 | 原模板页表单提交 | React 组件 + `react-dropzone` |

### 3.2 HeroUI v3 的最终定位

HeroUI v3 在本方案中不是局部引入，而是作为统一 UI 基础层：

- 所有通用按钮、卡片、输入框、下拉、表格、模态框、进度条、骨架屏统一使用 `@heroui/react`
- 主题样式统一基于 `@heroui/styles`
- 保留 `Lucide React` 作为图标体系
- 原 `components/ui/*` 中仅保留与项目强绑定的封装逻辑；通用基础控件不再继续扩展 `shadcn/ui`

HeroUI v3 的采用依据与保留结论：

- 无需 Provider 组件
- CSS-based 动画，无需额外引入 `framer-motion`
- 复合组件模式，适合页面级布局和业务组件封装
- 基于 React Aria，满足无障碍支持需求
- 主题变量支持 `oklch`

## 四、架构设计

### 4.1 前端分层架构

最终采用分层式 React 前端架构：

```text
app/                    路由与页面编排层
components/             组件层
lib/api/                接口访问层
lib/hooks/              业务行为复用层
lib/stores/             全局客户端状态层
lib/types/              类型层
lib/utils/              工具层
public/                 静态资源层
```

职责边界如下：

- `app/`：路由入口、页面级数据获取、页面布局组合
- `components/`：HeroUI 基础组件组合、业务组件封装、图表组件、AI 工具组件
- `lib/api/`：HTTP 请求、接口封装、请求取消、错误处理
- `lib/hooks/`：认证、SSE、轮询、SWR 数据获取、防抖、持久化
- `lib/stores/`：跨页面共享状态，主要为认证和商品相关轻量全局状态
- `lib/types/`：接口响应类型、业务模型类型
- `lib/utils/`：格式化、校验、常量、类名合并

### 4.2 页面架构原则

每个业务页面统一分为四层：

1. `page.tsx` 负责路由入口与页面骨架
2. 页面内部组合 `layout` 组件与业务组件
3. 数据访问走 `lib/api/*` 与 `lib/hooks/*`
4. 交互状态收敛到组件局部状态、SWR 缓存、Zustand Store

### 4.3 服务端与客户端职责划分

最终遵循以下原则：

- 首屏可直接获取的数据优先在 Server Components 获取
- 需要轮询、流式响应、用户主动交互的数据在 Client Components 获取
- 所有后端 API 调用统一经 `lib/api/*` 管理

适用场景：

- `dashboard` 初始统计数据：优先并行服务端获取
- `test1` SSE 文案生成：客户端处理
- `test2` / `test3` 任务轮询：客户端处理
- 商品 CRUD：客户端调用 API，SWR 刷新缓存

## 七、路由策略

### 7.1 App Router 方案

最终统一采用 `Next.js App Router`：

- 页面路由通过 `app/**/page.tsx` 管理
- 页面级加载态通过 `loading.tsx`
- 页面级错误态通过 `error.tsx`
- 404 通过 `not-found.tsx`

### 7.2 路由命名规范

路由命名以“业务域 + 子功能”组织，不再沿用 `test1`、`test2`、`test3` 这类临时命名：

- `test1.html` → `/ai-tools/copywriting`
- `test2.html` → `/ai-tools/image`
- `test3.html` → `/ai-tools/video`
- `setting.html` → `/live`
- `note.html` → `/marketing/notes`

### 7.3 代理与 rewrite 策略

现有 `frontend/next.config.js` 中已确认以下代理规则，最终方案继续保留：

```js
async rewrites() {
  return [
    {
      source: '/login',
      destination: '/'
    },
    {
      source: '/api/login',
      destination: 'http://localhost:3002/api/login'
    },
    {
      source: '/api/:path*',
      destination: 'http://localhost:5000/api/:path*'
    }
  ];
}
```

说明：

- `Login Server` 仍通过 `/api/login` 代理到 `3002`
- 主业务 API 仍通过 `/api/:path*` 代理到 `5000`
- `5001` 数据看板服务在重构后应逐步收敛为统一 API，由前端经 `lib/api/dashboard.ts` 消费

### 7.4 认证路由策略

认证路由保留现有登录入口语义：

- 登录页：`/`
- 兼容别名：`/login` -> `/`
- 登录成功后跳转：`/home`

## 九、状态管理方案

### 9.1 状态管理总原则

最终采用“局部状态 + SWR + Zustand”三层组合：

- 组件局部状态：表单输入、弹窗开关、筛选面板状态
- SWR：服务端数据缓存与刷新
- Zustand：跨页面共享但不适合放 URL 的轻量全局状态

### 9.2 Zustand 使用范围

保留为轻量全局状态：

- `lib/stores/auth-store.ts`
- `lib/stores/product-store.ts`

建议职责：

- `product-store.ts`：当前商品筛选上下文、局部缓存或跨页面编辑上下文

不建议用 Zustand 承载：

- 可通过 SWR 管理的服务端资源列表
- 可通过 URL 表达的筛选条件
- SSE 流式内容与轮询进度

### 9.3 SWR 使用范围

SWR 作为服务端数据获取的标准方案，适用于：

- 数据看板

关键文件：

### 9.4 URL 状态策略

筛选类状态通过 `nuqs` 或 query string 管理，适用于：

- `dashboard` 的 `timeRange`、`type`、`platforms`

## 十、数据获取与 Hook 方案

### 10.1 基础 HTTP 客户端

统一入口：`lib/api/client.ts`

最终必须具备的能力：

1. Token 注入
2. 统一错误处理
3. 请求取消
4. 必要时自动重试

保留的核心形式如下：

```ts
const API_BASE = process.env.NEXT_PUBLIC_API_URL || '';

export async function request<T>(
  endpoint: string,
  options?: RequestInit
): Promise<T> {
  const token = localStorage.getItem('token');

  const response = await fetch(`${API_BASE}${endpoint}`, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      ...(token && { Authorization: `Bearer ${token}` }),
      ...options?.headers
    }
  });

  if (!response.ok) {
    throw new Error(`API Error: ${response.statusText}`);
  }

  return response.json();
}
```

技术审查后的补充要求：

- 引入 `AbortController`
- 支持 `requestId`
- 对相同请求可取消旧请求

### 10.2 认证 Hook

文件：`lib/hooks/use-auth.ts`

最终要求：

- 本地读取 token
- 调用 `authApi.me()` 校验用户
- 登录后写入 token
- 登出后清空 token 和 store
- 页面跳转到 `/home` 或 `/`

### 10.3 看板数据 Hook

文件：`lib/hooks/use-dashboard.ts`

最终要求：

- 使用 SWR
- 稳定的 SWR key
- 支持 `refreshInterval`
- 支持错误重试
- 支持并行获取收入、利润、平台、销售数据

技术审查结论：

- 原 `['dashboard', params]` 的 key 不稳定
- 最终应使用序列化后的稳定 key

### 10.4 流式响应 Hook

文件：`lib/hooks/use-stream.ts`

最终要求：

- 处理 `test1` 对应文案流式输出
- 解析 `data: {...}` SSE 格式
- 支持中断、重置、错误处理

技术审查结论：

- 必须加入 `AbortController`
- 组件卸载时自动取消请求

### 10.5 任务轮询 Hook

文件：`lib/hooks/use-task-polling.ts`

最终要求：

- 支持图片/视频任务提交
- 轮询 `/api/tasks/{id}` 或兼容旧状态查询接口
- 管理 `pending / processing / completed / failed`
- 记录 `progress`、`result`、`logs`

技术审查结论：

- 必须具备轮询清理机制
- 失败时采用指数退避

## 十五、性能与稳定性策略

### 15.1 性能策略

最终保留以下性能结论：

1. 消除瀑布请求，使用 `Promise.all()` 或 `Promise.allSettled()`
2. 图表等重型组件全部动态导入
3. 数据获取统一走 SWR 缓存
4. 列表、图表等纯组件使用 `memo`
5. 复杂派生值使用 `useMemo`
6. 加载反馈优先骨架屏而非单一 spinner

### 15.2 关键性能目标

| 指标 | 目标 |
|------|------|
| First Load JS | `< 200KB` |
| LCP | `< 2.5s` |
| FID | `< 100ms` |
| CLS | `< 0.1` |
| Lighthouse Performance | `> 90` |

### 15.3 技术审查后的必选优化项

- `useStream` 增加 `AbortController`
- `useDashboard` 使用稳定 SWR key
- `useTaskPolling` 增加清理机制与指数退避
- HTTP 客户端支持请求取消
- 图表组件避免不稳定依赖导致重复渲染


## 十六、最终结论

本次最终重构技术方案的确定结论如下：

1. 前端统一迁移到 `frontend/` 下的 `Next.js App Router` 体系。
2. UI 基础层统一采用 `HeroUI v3`，不再延续 `shadcn/ui / Bootstrap` 双轨并存模式。
3. 样式体系统一升级为 `Tailwind CSS v4 + @heroui/styles + CSS 变量主题`。
4. 状态管理采用 `局部状态 + SWR + Zustand` 的组合方案。
5. 路由体系以业务域重构，保留原页面语义，废弃 `test1/test2/test3` 这类临时命名。
6. API 访问统一收口到 `lib/api/*`，迁移期兼容旧接口，目标逐步收敛为 REST 风格。
7. 组件、Hook、类型、Store、样式、加载边界、错误边界都已落实到具体目录与文件。
8. HeroUI v3 的引入不是表层替换，而是重构后的统一组件规范与工程基础。

因此，本文件作为后续前端重构实施、评审与拆分任务的基线方案。
