# 新前端约束指南

这份指南定义了当前项目的新前端接入基线。你后续新增页面、接入
其他同学的重构结果、或把旧页面迁入新结构时，必须同时满足这里的
目录约束、路由约束、布局约束、组件约束和接口边界约束。默认目标
不是“把页面跑起来”，而是在不改业务逻辑、不改接口契约的前提下，
把页面稳定接入当前 `frontend` 的 App Router、主布局和导航体系。

## 适用范围

这份指南适用于根目录下 `frontend` 的所有新页面、重构页面、兼容页
和导航接入工作，也适用于与 `aigc-server`、`flask-server`、旧模板
页面联动的前端迁移任务。

- 适用对象包括 `app` 路由页面、`components` 业务组件、导航配置、
  页面壳层、兼容重定向页和主题样式。
- 不适用对象是后端业务逻辑、数据库结构、推理链路、任务调度逻辑和
  非前端接口协议调整。

## 当前前端基线

当前项目的新前端基于 Next.js App Router、React 19、HeroUI 3 和
Tailwind CSS 4。你在接入新页面前，必须先按现状接入，不要先引入
新的技术栈、状态管理方案或组件体系。

- 前端主目录是 `frontend/`。
- 根布局在 `frontend/app/layout.tsx`，统一注入全局样式和主题提供器。
- 主题提供器在 `frontend/components/theme/theme-provider.tsx`，当前
  使用 `next-themes`，并以 `class` 模式切换主题。
- 主布局组件是 `frontend/components/layout/main-layout.tsx`。
- `(main)` 路由组和 `ai-tools` 路由当前都挂载到 `MainLayout`。
- 导航配置中心是 `frontend/components/layout/nav-config.ts`。
- 导航渲染和激活态逻辑在
  `frontend/components/layout/nav-items.tsx`。

## 接入总原则

后续所有页面接入必须先满足架构一致性，再考虑页面视觉还原。你必须
优先复用现有布局、路由和组件边界，不能为了省事重新造一套页面壳层。

- 必须优先接入现有 `MainLayout`，不要在页面内再造侧边栏和顶栏。
- 必须优先接入现有导航管理，不要在页面组件里写分散的导航判断。
- 必须保持现有业务调用链不变，不能在页面重构时顺手改接口名、请求
  体、返回结构或轮询策略。
- 必须把兼容逻辑收口到路由层或页面壳层，不要把兼容判断散落到业务
  组件内部。
- 必须把样式调整控制在展示层，不要通过改状态流或数据流来“修 UI”。

## 目录与文件归属

目录归属必须稳定，否则后续页面会继续出现“页面能跑，但导航、布局、
主题、滚动、重定向都不统一”的问题。新页面先定归属，再开始写组件。

- 正式页面路由放在 `frontend/app/` 下。
- 需要统一后台壳层的页面，放入 `frontend/app/(main)/` 或其他显式
  接入 `MainLayout` 的路由层。
- `ai-tools` 正式页面当前使用
  `frontend/app/ai-tools/layout.tsx` 挂载 `MainLayout`，后续同类页面
  继续沿用这一做法。
- 页面级组合逻辑放在 `frontend/app/**/page.tsx`，不要把大段业务逻辑
  直接堆在路由文件里。
- 可复用业务组件放在 `frontend/components/` 下，并按业务域分目录。
- 导航配置只能集中维护在
  `frontend/components/layout/nav-config.ts`。
- 导航交互和激活态逻辑只能集中维护在
  `frontend/components/layout/nav-items.tsx`。
- 兼容重定向页保留最小逻辑，只负责 `redirect(...)`，不要承载任何
  业务组件。

## 路由约束

路由是后续接入工作的第一约束。只要一个页面被定义为“新前端正式页”，
它就必须有唯一正式 URL，并且旧入口只能做兼容跳转，不能继续和正式
 页并行演进。

- `ai-tools` 是当前工作台正式路由域。
- 文案、图片、视频三条工作台路由分别是：
  ` /ai-tools/textgenerate`
  ` /ai-tools/image`
  ` /ai-tools/video`
- 旧 `workspace/*` 路由只保留兼容职责，不再承载正式页面实现。
- 新增重构页时，先确认它属于哪个正式路由域，再决定是否需要旧地址
  兼容页。
- 不要同时维护两套正式入口，比如同时把 `workspace/*` 和
  `ai-tools/*` 都当成主入口。
- 不要在组件内部写基于旧路径的业务判断，路径兼容必须在路由层完成。

## 布局与滚动约束

当前后台布局已经收口到固定视口框架。后续页面接入必须服从这一滚动
模型，否则会重新出现整页滚动、侧边栏跟滚、页面内部抢滚动等问题。

- `MainLayout` 是唯一后台主壳层。
- 整个后台界面固定在视口高度内。
- 左侧侧边栏保留自己的滚动能力。
- 顶栏固定在右侧工作区顶部。
- 右侧主内容区由布局层的 `main` 负责主滚动。
- 页面外层不要再额外声明会和主布局抢职责的全屏滚动容器，例如新的
  `h-screen`、`overflow-auto` 或二次 `fixed` 壳层。
- 只有当页面内部存在明确的局部滚动区域，例如聊天消息区、历史列表
  或表格体，才允许在局部组件中使用内部滚动。
- 页面接入后，如果出现“滚不动”或“双滚动条”，优先排查页面外层容器，
  不要先改 `MainLayout`。

## 导航管理约束

导航必须是“配置驱动 + 集中渲染”。后续接入新页面时，不要把导航标题、
图标、激活态和重定向逻辑分散写在多个文件里。

- 菜单结构、图标、跳转地址统一维护在
  `frontend/components/layout/nav-config.ts`。
- 当前路径匹配、分组展开、折叠态展示和下拉动作统一维护在
  `frontend/components/layout/nav-items.tsx`。
- 新增页面要进左侧导航时，先补 `nav-config.ts`，再补
  `nav-items.tsx` 的激活态和分组判断。
- 不要在页面组件里自行高亮导航，也不要在业务组件里写
  `router.push` 来模拟左侧菜单行为。
- 旧地址兼容跳转不等于导航入口。导航必须始终指向正式路由。

## React 约束

React 层的重构目标是保持组件边界清晰、状态来源单一、异步副作用可控。
你必须优先遵循 React 19 与当前代码风格，而不是引入额外抽象来追求
“更现代”的写法。

- 页面文件只负责路由接入和页面级组装。
- 业务流程、请求状态、轮询、选择器状态和结果展示放入对应业务组件。
- 优先使用函数组件和 Hooks。
- 只在确实需要时引入 `useMemo`、`useCallback` 和 `startTransition`。
  不要为了“看起来专业”而机械包裹所有值和函数。
- 异步请求必须有明确的加载态、异常态和清理逻辑。
- 存在轮询、流式响应、定时器或 `AbortController` 的页面，必须在组件
  卸载时清理副作用。
- 不要在渲染阶段直接访问 `window`、`localStorage` 或浏览器专属 API。
  这类逻辑必须放进客户端组件和 `useEffect`。
- 不要把后端返回数据结构二次发明成另一套前端协议，优先做最薄的一层
  适配。

## HeroUI 约束

HeroUI 是当前新前端的组件基线。后续页面接入要优先复用 HeroUI 和现有
 设计 token，不要混入另一套组件系统，也不要在同一层重复造轮子。

- 优先使用 `@heroui/react` 组件完成按钮、卡片、输入、选择器、下拉、
  Toast 和滚动容器等通用 UI。
- 优先沿用全局 token、`globals.css` 和现有 `className` 风格，不要在
  单页里写大段孤立的硬编码设计体系。
- 需要组合型布局时，先组合 HeroUI 组件和 Tailwind 工具类，再决定
  是否封装新组件。
- 能用语义化属性解决的交互，不要额外写无意义包裹层。
- 组件的可访问名称、`aria-*` 属性和可聚焦行为必须完整保留。
- 不要为了还原旧静态页面，把 HeroUI 组件退化成只剩外观的 `div`。

## 样式与主题约束

样式改造必须服从现有全局 token、布局骨架和主题注入方式。你可以做
局部视觉调整，但不能在新页面里重新定义一套全局背景、字体和配色规则。

- 全局样式入口是 `frontend/app/globals.css`。
- 主题切换基于 `next-themes` 的 `class` 模式。
- 新页面必须先复用现有背景、边框、圆角、阴影和文本色 token。
- 不要在页面根节点写覆盖全局背景的强耦合样式，除非该页面被明确设计
  为独立专题页。
- 不要通过固定高度和绝对定位强行拼出后台壳层。
- 样式问题优先在页面容器层修复，不要通过改业务组件数据结构来回避布局
  问题。

## 业务逻辑边界

这是后续所有重构工作的硬红线。前端接入和重构默认只处理展示层、路由、
壳层、导航和轻量适配，不得擅自改变现有业务行为。

- 不得修改接口路径。
- 不得修改请求方法。
- 不得修改请求体字段名。
- 不得修改后端返回结构约定。
- 不得修改任务轮询节奏、流式消费协议和保存逻辑，除非你先得到明确确认。
- 不得把原本在页面组件中的业务流程迁移到后端，或把原本后端提供的
  结果判定搬到前端重算。
- 如果你不确定某个字段、状态或接口是否属于业务逻辑，先汇报，再执行。

## 现有调用链路基线

后续接入和重构必须以当前真实调用链为准，而不是以设计稿、口头描述或
旧模板文件名为准。下面列的是当前三条 `ai-tools` 链路的前端接入基线。

### 文案智造器

文案页面当前是一个前端流式生成页。你在做布局接入、视觉还原或组件替换
时，必须保留它的流式响应、表单提交流程和草稿保存流程。

- 页面路由是 `/ai-tools/textgenerate`。
- 页面入口组件在 `frontend/components/ai-tools/textgenerate/`。
- 生成接口是 `/generate_xiaohongshu_stream_post`。
- 产品信息提交接口是 `/api/product-info`。
- 表单数据提交接口是 `/submit-form-data`。
- 流式消息、问题提取、保存草稿和已保存内容列表都属于现有业务流程，
  不得在重构时删除或改协议。

### 营销图创作

图片页面当前包含运行时配置、分类商品选择、任务创建、任务轮询、历史
读取和结果保存这条完整链路。页面重构只能改展示与布局，不能改这条链路。

- 页面路由是 `/ai-tools/image`。
- 页面入口组件在 `frontend/components/ai-tools/image/`。
- 运行时配置接口是 `/api/runtime-config/image`。
- 分类列表接口是 `/api/oss/categories`。
- 分类商品接口是 `/api/oss/products_by_category`。
- 生成接口是 `/api/generate_img2img`。
- 任务状态接口是 `/api/check_task_status/{taskId}`。
- 历史记录接口是 `/api/tasks/image`。
- 结果保存接口是 `/api/save_generated_content`。

### 短视频智造

视频页面当前和图片页面共享一部分商品与配置链路，但生成、历史和结果
结构独立。后续重构时，不要因为页面相似就把两条链路粗暴合并。

- 页面路由是 `/ai-tools/video`。
- 页面入口组件在 `frontend/components/ai-tools/video/`。
- 运行时配置接口是 `/api/runtime-config/image`。
- 分类列表接口是 `/api/oss/categories`。
- 分类商品接口是 `/api/oss/products_by_category`。
- 生成接口是 `/api/generate_video`。
- 任务状态接口是 `/api/check_task_status/{taskId}`。
- 历史记录接口是 `/api/tasks/video`。
- 结果保存接口是 `/api/save_generated_content`。

## 新页面接入流程

后续新增一个前端重构页时，按下面顺序执行。你必须先定正式归属和边界，
再写页面代码，这样才能避免返工。

1. 确认该页面的正式路由域，以及是否需要旧地址兼容页。
2. 确认该页面是否必须进入 `MainLayout`。
3. 确认导航是否需要新增入口、分组或激活态规则。
4. 盘点现有接口、请求体、返回体和轮询逻辑，标出不可变更项。
5. 先完成页面壳层接入，再迁移业务组件。
6. 只在展示层做必要适配，避免顺手重写业务逻辑。
7. 接入完成后，验证路由跳转、导航高亮、滚动行为、加载态、异常态和
   实际接口调用。

## 重构时的禁止事项

下面这些做法在当前项目里风险最高。除非你已经和需求方明确对齐，否则
不要执行。

- 不要新造第二套后台壳层。
- 不要在页面内复制一套假侧边栏和假顶栏。
- 不要把旧页面路径继续当正式导航入口。
- 不要在改布局时顺手改接口。
- 不要在改视觉时顺手改组件状态机。
- 不要把兼容逻辑写进业务组件。
- 不要把一个页面的滚动问题通过全局布局回退来“快速解决”。
- 不要混用多套 UI 组件库。
- 不要在未确认前提下修改深色模式策略。

## 页面验收清单

每次接入或重构一个页面后，你都必须至少完成一次人工验收。这个验收
清单用于判断页面是否真正接入完成，而不是只完成了视觉还原。

- 正式 URL 能直接访问。
- 旧 URL 如果需要兼容，能够正确重定向。
- 左侧导航能正确跳转并高亮。
- 页面进入后不重复渲染侧边栏和顶栏。
- 右侧内容区能够独立滚动。
- 页面加载、错误、空状态和处理中状态正常显示。
- 实际请求仍然命中原有接口。
- 请求体和返回体解析没有变化。
- 轮询、流式输出和保存能力没有退化。
- `pnpm build` 能通过。

## 后续维护建议

这份指南的目标不是一次写完后长期不动，而是作为新前端的准入规则持续
维护。只要主布局、导航策略、主题方案或正式路由域发生变化，你就要同步
更新这份文档。

- 新增一个正式路由域时，补充这里的路由约束和接入示例。
- 新增一个共享布局时，明确它和 `MainLayout` 的边界。
- 新增一个可复用业务模块时，补充它的目录归属和不可变更接口边界。
- 如果未来正式启用深色模式规范，再单独补一节主题与暗色约束。

## 下一步

后续任何前端页面接入、布局重构或旧页面迁移，都应先对照这份指南确认
归属、边界和不可变更项，再开始改代码。这样能最大限度减少路由反复、
导航漂移、样式冲突和接口回归问题。
