# 后端重构方案

## 1. 项目现状

当前项目已经从旧的 UE5/Fay 表现层迁移到 Live2D，但后端仍保留大量 Fay 时代形成的耦合方式。最典型的问题是：

- `gui/aigc_server.py` 同时承担页面路由、营销内容、直播接口、ASR、聊天、音频文件、WebSocket 状态、OSS 操作
- 数字人主链虽然已经转到 `avatar_runtime -> avatar_core`，但接口层仍然和具体实现耦合
- 配置、TTS、ASR、历史消息、WebSocket、兼容层分散在多个模块里
- `6000` 与 `5000` 之间曾长期存在重复接口，后续维护容易漂移

## 2. 当前调用链梳理

### 2.1 HTTP 主链

当前默认 HTTP 入口：

- `main.py`
  - 加载环境变量和配置
  - 初始化数据库与 WebSocket
  - 按需启动：
    - `gui/login_server.py`
    - `gui/app.py`
    - `gui/aigc_server.py`
    - `gui/flask_server.py`

### 2.2 直播主链

直播相关接口主要在 `gui/aigc_server.py`：

- `/api/get-data`
  - 读取 `config.json` / `system.conf`
  - 拼装音色列表
- `/api/submit`
  - 保存配置
- `/api/start-live`
  - 启动 WebSocket 服务
  - 启动数字人运行时
- `/api/stop-live`
  - 停止数字人运行时
- `/api/chat`
  - 构造 `Interact`
  - 调用 `avatar_runtime.get_instance().on_interact(...)`
- `/api/asr`
  - 接收上传音频
  - 必要时转 WAV
  - 调用本地 Qwen3-ASR 服务
- `/api/get-msg`
  - 从消息库读取会话记录
- `/v1/chat/completions`
  - 将 OpenAI 风格请求转成内部 `Interact`
  - 交给数字人核心处理

### 2.3 数字人内部调用链

当前数字人主链已经是：

`HTTP Route -> avatar_runtime -> core/avatar_core.py -> LLM/TTS/WebSocket/DB`

其中：

- `avatar_runtime.py`
  - 控制运行时实例生命周期
- `core/avatar_core.py`
  - 负责用户消息处理
  - 调用 QA / LLM
  - 写入消息库
  - 通过 WebSocket 推前端状态与回复
  - 调用 TTS 输出音频
- `core/recorder.py`
  - 负责语音输入链路
  - 继续通过 `avatar_core` 回到主对话链

### 2.4 Live2D 前端接收结构

前端直播页当前主要消费两类数据：

- WebSocket 推送
  - `panelMsg`
  - `panelReply`
  - `liveState`
  - `is_connect`
  - `remote_audio_connect`
- HTTP 拉取
  - `/api/get-msg`
  - `/api/latest-audio`
  - `/api/ws-status`

## 3. 当前存在的问题

### 3.1 路由层过重

`gui/aigc_server.py` 过于庞大，路由函数直接操作：

- 配置
- 运行时
- WebSocket
- 数据库
- ASR 请求
- 音频文件

这会导致：

- 代码可读性差
- 资源生命周期分散
- 难以控制内存和对象复用

### 3.2 主链编排和具体实现耦合

直播接口直接依赖：

- `avatar_runtime`
- `wsa_server`
- `content_db`
- `member_db`
- `requests.post`

这意味着后面切换：

- ASR 实现
- TTS 实现
- 数字人驱动实现

都会牵动路由层。

### 3.3 Fay 旧逻辑仍有兼容残留

当前已经完成主链去 Fay 化，但仓库里仍保留兼容入口：

- `fay_booter.py`
- `core/fay_core.py`
- `fay_url`
- `fay.db`

这些残留现在已经被压缩到兼容层，但仍需要继续收敛，避免新功能误接到旧命名上。

### 3.4 资源管理分散

历史问题主要包括：

- 重模块在入口阶段一次性导入
- 外部 HTTP 调用使用一次性 `requests.post`
- 音色列表每次路由都重新拼装
- WebSocket、数字人运行时、配置加载没有统一 manager

这会带来：

- 不必要的内存占用
- 连接建立成本增加
- 运行状态判断分散

## 4. 推荐的新调用链

建议将直播主链统一为：

`Flask Route -> Service -> Manager -> Adapter -> Runtime/Core`

具体如下：

### 4.1 HTTP 接口层

- 只负责：
  - 收参
  - 调用服务层
  - 返回 JSON / 文件响应

### 4.2 服务层

- 负责完整编排：
  - 配置读取
  - 启停 runtime
  - 会话查询
  - ASR 上传处理
  - OpenAI 风格协议适配

### 4.3 Manager 层

- 负责资源复用：
  - 运行时状态
  - WebSocket 启停
  - HTTP Session 复用
  - 音色列表缓存

### 4.4 Adapter 层

- 负责第三方或兼容系统接入：
  - Qwen3-ASR
  - Legacy Fay compatibility
  - 后续可扩展 TTS / LLM adapter

## 5. 本轮已落地的目录结构

本轮已经落地以下新结构，并接管直播主链：

```text
backend/
  config/
    runtime_config.py
  routes/
    live_routes.py
  services/
    live_service.py
  managers/
    runtime_manager.py
    voice_catalog.py
    http_session_manager.py
  adapters/
    asr/
      qwen3_adapter.py
    legacy/
      fay_adapter.py
  schemas/
    live_dto.py
```

## 6. 本轮已完成的重构动作

### 6.1 路由层瘦身

`gui/aigc_server.py` 的以下接口已改为委派到新 route/service 层：

- `/api/get-data`
- `/api/submit`
- `/api/start-live`
- `/api/stop-live`
- `/api/get-msg`
- `/api/get-member-list`
- `/api/chat`
- `/api/asr`
- `/api/latest-audio`
- `/api/ws-status`
- `/v1/chat/completions`
- `/api/send/v1/chat/completions`

### 6.2 运行时管理集中化

新增 `backend/managers/runtime_manager.py`：

- 统一管理 WebSocket bootstrap
- 统一管理 `avatar_runtime` 启停
- 统一查询 live 状态

### 6.3 ASR 调用适配层

新增 `backend/adapters/asr/qwen3_adapter.py`：

- 统一处理上传音频
- 统一执行 WAV 转换
- 统一调用 Qwen3-ASR 服务
- 通过 `requests.Session` 复用连接

### 6.4 资源复用优化

新增：

- `backend/managers/http_session_manager.py`
  - 复用外部 HTTP 连接，减少重复建立连接
- `backend/managers/voice_catalog.py`
  - 缓存音色列表，减少每次请求重复构建

### 6.5 Fay 兼容隔离

新增：

- `backend/adapters/legacy/fay_adapter.py`

用于把旧 Fay 风格兼容入口集中在 adapter 层，避免继续污染主链。

## 7. 内存与显存优化建议

### 7.1 已做优化

- 主入口按需导入 HTTP 服务，减少启动时重模块常驻
- `6000` 兼容层不再重复加载直播主业务依赖
- 直播主链统一走 `avatar_runtime`
- ASR 外部请求改为 Session 复用
- 音色列表增加小型缓存
- 数据结构解析集中到 DTO，减少路由层临时对象散落

### 7.2 下一步建议

建议继续做这几项低风险优化：

1. 给 TTS 增加 `tts_manager`
   - 统一持有 TTS client
   - 避免后续多个入口重复创建合成客户端

2. 给 LLM 增加 `llm_service`
   - 把 `core/avatar_core.py` 里的 LLM 选择逻辑继续抽出去
   - 让 `avatar_core` 更聚焦于“对话编排 + 状态广播”

3. 把 `core/avatar_core.py` 再拆成：
   - `dialog_service`
   - `speech_service`
   - `emotion_service`

4. 为样本音频目录增加清理策略
   - 避免 `samples/` 长期积累过多 wav 文件

5. 为大对象提供显式关闭点
   - TTS client
   - 外部 Session
   - 可能的模型连接句柄

## 8. 推荐的后续迁移顺序

为保证低风险，建议按下面顺序继续推进：

1. 先继续拆 `core/avatar_core.py`
2. 再补 `tts_manager` 与 `llm_service`
3. 再把 `gui/aigc_server.py` 中非直播业务继续拆到独立 route/service
4. 最后把历史兼容命名进一步收敛，只保留必要的别名层

## 9. 当前结论

当前项目已经具备了一个较清晰的新主链：

`Live2D Frontend -> Flask Route -> Live Service -> Runtime Manager / ASR Adapter -> avatar_runtime -> avatar_core`

这意味着：

- 新链路默认不依赖 Fay
- Fay 仅作为兼容层存在
- Flask 框架保持不变
- 接口协议基本保持兼容
- 后续可以在不推倒重来的前提下继续渐进式重构
